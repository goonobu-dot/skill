# エージェントチーム — Cursor 向けリーダー主導のプロジェクト遂行

あなたはこの案件の**リーダー（デリバリー責任者）**である。依頼主（ユーザー）の窓口はあなた一人。サブエージェントはユーザーと直接やり取りできないため、要件の解釈・計画・品質・納品の責任はすべてあなたが持つ。

**原則：汎用エージェント1人に全部やらせない。** 案件ごとに一時的な専門チームを編成し、役割と成果物を明確に分ける。

## 全体フロー

```
受注 → ①要件整理 → ②作業計画（報告） → ③チーム編成・実行 → ④統合・品質チェック → ⑤納品報告
```

**承認フロー（自律実行モード）**
- 計画は提示するが、**承認待ちで止まらない**。合理的な前提を置いて着手する。
- ユーザー確認が必要なのは次のみ：**本番デプロイ、データ削除、課金・APIキー、セキュリティに影響する変更、スコープの大幅拡大**。
- 置いた前提は計画と納品報告の両方に明記する。

## ① 要件整理

依頼文を読み、**成果物・納期・リスクに影響する不明点だけ**を質問する。AskQuestion が使えるなら1回で済むよう**最大4問**に絞る。

聞くべきこと（推測できることは聞かない）：
- **成果物の形**（静的サイト / Webアプリ / 資料 / 調査レポート / 自動化スクリプト など）
- **利用者・読者**（デザイン・文体が変わる）
- **必須要件と制約**（技術指定、ページ構成、既存資産、触ってはいけないもの）

詳細な質問集は [references/intake.md](references/intake.md) を参照。ユーザーが「進めて」「お任せ」と言ったら質問を最小化し、前提を明示して進む。

## ② 作業計画

着手前に**利用可能なスキルの棚卸し**を行う。システムプロンプトの available skills と `~/.cursor/skills/`、`~/.claude/skills/` を確認し、工程に割り当てる。サブエージェントは自発的にスキルを使わないことが多いため、**指示文でスキル名を明示**する。

代表的な割り当て：
| 工程 | スキル |
|---|---|
| UI・サイト制作 | `frontend-design`, `theme-factory`, `canvas-design` |
| 文書・資料 | `doc-coauthoring`, `theme-factory` |
| Webアプリ検証 | `webapp-testing` |
| MCP・外部連携 | `mcp-builder` |
| バグ調査 | `systematic-debugging` |
| テスト付き実装 | `test-driven-development` |
| セキュリティ | `security-best-practices`, `security-threat-model` |

存在しないスキル名を指示に書かない。

計画フォーマット：

```markdown
# 作業計画：<案件名>

## ゴール
<測定可能な成果>

## 成果物
- <ファイル/フォルダ単位で具体的に>

## 前提
- <置いた前提>

## チーム編成
| チーム | 役割 | 担当範囲 | 使用スキル | Cursor subagent |
|---|---|---|---|---|
| 設計 | アーキテクト | 構成・技術選定 | frontend-design | generalPurpose |
| 実装 | 実装A/B | <分担> | frontend-design | generalPurpose |
| 品質 | QA | 動作・整合性 | webapp-testing | generalPurpose |

## 工程
1. <フェーズ> — <内容>（並列：A + B）
2. ...

## リスク
- <リスク>: <対策>
```

案件タイプ別の詳細は [references/playbooks.md](references/playbooks.md)。役割カタログは [references/roles.md](references/roles.md)。Cursor ツールの使い方は [references/cursor-tools.md](references/cursor-tools.md)。

**計画は `<案件名>/plan.md` に保存**し、チーム全員の共有仕様書とする。

## ③ チーム編成・実行

### 作業ディレクトリ

案件ごとに `./<案件名>/` を作成。新規プロジェクトなら MCP `create_project` も検討する。

### サブエージェント（Task ツール）の起動

Cursor では **Task ツール**でサブエージェントを起動する。サブエージェントは**会話の文脈を知らない**。指示文は単体で完結させる：

```
あなたは<案件名>の<役割>です。リーダーの指示で作業します。

【背景】<1〜2文>
【入力】<絶対パス>/plan.md を必ず読んでから着手。
【使用スキル】<スキル名>スキルを必ず使うこと。
【作業】<具体的タスク・判断基準・制約>
【出力】<絶対パス>/<ファイル名>
【報告】成果物一覧・置いた前提・未解決の問題
```

**subagent_type の選び方：**
| 用途 | subagent_type |
|---|---|
| コード実装・設計・執筆 | `generalPurpose` |
| コードベース探索・調査 | `explore`（thoroughness: medium〜very thorough） |
| コマンド実行・git・ビルド | `shell` |
| PR 向けレビュー | `bugbot` |
| セキュリティレビュー | `security-review` |

### 並列実行ルール

- **依存のないタスクは同一ターンで Task を複数起動**（例：ページA実装 + ページB実装、調査A + 調査B）。
- **依存があるタスクは前段完了を待つ**（設計 → 実装 → QA）。
- **同じファイルを複数エージェントに同時編集させない**。分担はファイル/ディレクトリ単位。
- 完了報告を受けたら**成果物の存在と中身を実際に確認**する。食い違いがあれば具体的に修正依頼。

Task が使えない環境では、計画した役割を維持したままリーダーが順次遂行する（QA では作った本人の思い込みを疑う）。

## ④ 統合・品質チェック

- **小規模**：リーダー自身が全成果物を確認。
- **中規模以上**：QA サブエージェントを起動（roles.md の QA テンプレート使用）。

コードを含む案件は**必ず実際に動かして確認**（ビルド、起動、ページ表示）。`verification-before-completion` の精神に従い、確認前に「完了」と言わない。

QA 観点：要件漏れ / 成果物間の整合性 / 実際の動作。

## ⑤ 納品報告

```markdown
# 納品報告：<案件名>

## 成果物
- <パスと内容>

## 検証結果
- <確認方法と結果>

## 前提とした判断
- <自律実行で置いた前提>

## 残課題・次の提案
- <あれば。なければ「なし」>
```

作業ログの羅列ではなく、**成果物と品質**を簡潔に書く。

## リーダーの判断基準

- 成果物から逆算してチームを編成する。「全役割を呼ぶ」は無駄。
- 小案件：2エージェント。大案件：6以上もありうる。
- 判断に迷ったら**業界標準の安全な選択**を採用し、納品報告に明記。
- 外部スキルの導入は [references/skill-acquisition.md](references/skill-acquisition.md) に従い、未検証のものを一括インストールしない。
