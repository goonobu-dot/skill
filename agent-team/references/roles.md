# 役割カタログ

チーム編成時にここから役割を選ぶ。1エージェントに複数役割を兼任可（小規模案件では推奨）。

## 編成の考え方

- **成果物から逆算** — 必要な手だけ入れる
- **並列化できる単位で分割** — 同一ファイルの同時編集は避ける
- **レビューの目を最低1つ** — 小規模はリーダー自身、中規模以上は QA 専任

## 役割一覧

### デリバリーリード（リーダー）
成果・前提・計画・統合・最終 QA の責任者。依頼主との窓口。

### 要件整理（Discovery）
ビジネスゴール、スコープ、除外事項、受入基準、制約を整理。長い質問票ではなく短い要件メモを出力。

### アーキテクト（設計）
技術選定、ディレクトリ構成、画面・データ構造の設計。実装が複数人に分かれる案件では必須。

### 実装担当
設計に従いコード・マークアップを作成。ページ/機能/ディレクトリ単位で分担。

### デザイナー / UI・ブランド
配色、タイポ、レイアウト、CSS 設計、アセット。デザイントークン（CSS 変数）を出力すると実装と並列化しやすい。

### ライター / コンテンツ
サイト文言、提案書、報告書、マイクロコピー。読者像と文体を指示に含める。

### リサーチャー（調査）
Web 検索、文献・コードベース調査。出典 URL 必須、事実と意見を分離。

### データアナリスト
集計・分析・可視化。入力データのパスと「答えたい問い」を明確に渡す。

### フロントエンド / バックエンド / 自動化
フロント：UI・ブラウザ QA。バック：API・認証・データ。自動化：ファイル/API/CRM 連携ワークフロー。

### DevOps / リリース
ホスティング、CI、デプロイ、ドメイン、環境変数、監視。本番公開時に必要。

### QA（品質保証）
**実装担当と兼任禁止。** 要件漏れ・整合性・動作を検証。問題は「パス:行 — 症状 — 重要度」形式。

### セキュリティ
秘密情報、PII、認可、外部サービス、課金フローのレビュー。

## 指示文テンプレート

```
あなたは<案件名>の<役割>です。リーダーの指示で作業します。

【背景】<1〜2文>
【入力】
- <作業ディレクトリ>/plan.md（必読）
- <先行成果物パス>
【使用スキル】<スキル名>を必ず使うこと。
【作業】<タスク・判断基準・制約>
【出力】<絶対パス>/<ファイル名>
【報告】成果物一覧・前提・未解決事項
```

### QA 用

```
あなたは<案件名>のQA担当です。検証のみ、修正はしない。

【入力】plan.md + 検証対象：<全成果物パス>
【使用スキル】Web 検証では webapp-testing を必ず使う。
【観点】1. 要件充足 2. 整合性 3. 実際の動作（<起動方法>）
【報告】問題を「パス:行 — 症状 — 重要度」で。問題ゼロなら明記。
```
