# 案件タイプ別プレイブック

デフォルトの進め方。ローカルのファイル・ツール・緊急度に合わせて調整する。

## ホームページ / LP 制作

**チーム例：** リーダー、要件整理、UX、UI/ブランド、コンテンツ、フロント、QA

**工程：**
1. 受付 — ゴール、読者、ページ数、参考サイト、コンバージョン
2. 構成 — サイトマップ、ファーストビュー、セクション、CTA、フォーム
3. コンテンツ — 文案、画像要件、連絡先、メタデータ
4. 実装 — レスポンシブ、ナビ、アクセシビリティ、フォーム
5. QA — モバイル/デスクトップ、リンク、フォーム、誤字
6. 納品 — ファイルまたは URL、残タスク、公開チェックリスト

**よくあるリスク：** 画像未準備、オファー不明、ドメイン/ホスティング未確保、フォーム送信先なし

**技術選定（指定なし）：** 静的 HTML/CSS/JS を優先。レスポンシブ必須。

## システム / SaaS 開発

**チーム例：** リーダー、要件整理、プロダクト、UX、フロント、バック、DevOps、QA、セキュリティ

**工程：**
1. 要件 — ユーザー、ロール、ワークフロー、データ、連携、受入基準
2. アーキテクチャ — スタック、データモデル、API、認証、環境
3. プロトタイプ — ハッピーパスの動作確認
4. コア実装 — ワークフロー単位の縦割り、共通部分にテスト
5. 運用 — ログ、設定、エラー状態
6. リリース — デプロイ、監視、ロールバック、引き継ぎ

**よくあるリスク：** ロール曖昧、レガシー連携、認証/セキュリティ不足、本番データ

## 資料作成（提案書・報告書・スライド）

**チーム例：** リーダー、要件整理、リサーチャー、ライター、QA

**工程：**
1. 読者と目的の確定
2. 構成案（章立て）
3. 素材収集（調査メモがあれば並列）
4. 執筆
5. 整合性・誤字・数値の QA

**スキル：** `doc-coauthoring`（構成から共著）、`theme-factory`（スライド/資料のテーマ）

## 調査・分析

**チーム例：** リーダー、リサーチャー（観点ごとに分割可）、エビデンスレビュー、ストラテジー

**工程：**
1. 支援する意思決定と評価基準を定義
2. ソース収集（公式/公開を優先）
3. 事実・前提・推定・意見を分離
4. 基準に沿って比較
5. 推奨、注意点、次アクションを納品

## 業務自動化 / AI エージェント

**チーム例：** リーダー、要件整理、自動化、Data/AI、セキュリティ、QA

**工程：**
1. 現状の手作業と失敗点をマップ
2. 入出力、承認ゲート、リトライ、ログ、手動フォールバックを定義
3. 最小の信頼できるワークフローを構築
4. 実データでテスト
5. 運用手順書

**よくあるリスク：** 脆い UI 自動化、資格情報不足、無制限コスト、監査ログなし

## 保守 / 緊急修正

**チーム例：** リーダー、該当エンジニア、QA、（本番/データリスク時）セキュリティ

**工程：**
1. 再現または現象確認
2. 影響範囲とロールバック手段
3. 最小の安全な修正
4.  targeted テストまたは手動確認
5. 原因と再発防止を報告
